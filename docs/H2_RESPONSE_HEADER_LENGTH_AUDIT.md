# HTTP/2 Response Header Length Audit

Started: 2026-05-21
Branch: `research/poolslip-defensive-source-audit`

## Scope

This is a separate fixed-bug validation track discovered while checking the
small `release-1.31.0 -> nginx/1.31.1` source diff. It is not the Poolslip
demo/default-module track because the demo-like build does not include
`--with-http_v2_module`.

Objective: prove or reject the fixed HTTP/2 response-header length issue as an
ASAN-detectable memory-safety bug in a local lab, without developing an exploit
chain.

## Source Lead

Commit `58a7bc3406ac8b9dc0e0afafc69ba42df56009e3` adds length limits for very
large `Content-Type` and `Location` response fields in
`src/http/v2/ngx_http_v2_filter_module.c`.

The commit message says the old allocation undercounted the number of bytes
needed to encode very large field lengths. The expected vulnerable comparison
target is `release-1.31.0` built with `--with-http_v2_module`; the fixed
comparison target is `eff110885412737aec9b953067b6a670bffdbfa0`.

## Lab

Files added:

- `env/Dockerfile.h2-header`
- `env/nginx-h2-header.conf`
- `tools/poolslip_h2_upstream_probe.py`

Shared backend support added to `env/server.py`:

- `case=raw-gen&mode=huge-content-type&size=<n>`
- `case=raw-gen&mode=huge-location&size=<n>`

The lab raises proxy buffers so a local backend can provide multi-megabyte
special response headers to the HTTP/2 header filter.

The same lab also exposes `/h2-upstream`, which uses `proxy_http_version 2`
against a raw local HTTP/2 upstream on `127.0.0.1:19325`. This exercises
`ngx_http_proxy_v2_module`, a separate parser-heavy surface compiled only in
H2-enabled builds.

## Milestone Log

- 2026-05-21: Built `nginx-h2-header-1310-amd64-asan` from
  `release-1.31.0` with `--with-http_v2_module`, ASAN, and no gzip. Baseline
  `curl --http2-prior-knowledge http://127.0.0.1:19361/` returned
  `HTTP/2 200` and `Server: nginx/1.31.0`.
- 2026-05-21: Tested `huge-content-type` and `huge-location` generated by the
  local backend with sizes `2097152`, `3145728`, and `4194304` over HTTP/2
  prior knowledge. No ASAN report was produced and the worker stayed alive.
  Observed client behavior was timeout for large `Content-Type`, timeout or
  `INTERNAL_ERROR` for large `Location`.
- 2026-05-21: Source/runtime interpretation: `release-1.31.0` already enforces
  `NGX_HTTP_V2_MAX_FIELD` for generic response headers, so the `Location` case
  is rejected through the generic header path in this proxy setup. The special
  `Content-Type` path is the unfixed area, but the first live trigger did not
  cross an ASAN redzone, matching the upstream commit note that other headers
  can leave enough spare encoded-buffer space in normal module paths. Next
  useful refinement would be reducing encoded slack, for example by varying
  value byte composition and response headers, rather than treating the first
  timeout as proof of a crash.
- 2026-05-21: Added `fill=<char>` support for generated huge `Content-Type` and
  `Location` values, rebuilt the vulnerable H2 ASAN lab, and tested
  `huge-content-type` with `~`, `}`, and `|` fillers at `3145728` and
  `4194304` bytes. These characters reduce or remove HPACK Huffman savings for
  the oversized value. Result remained negative: client-side timeout with
  `code=200`, no ASAN report in Docker logs, no worker crash, and no NGINX
  error-log finding. This confirms the source-level fixed bug is real but has
  not yet been turned into an ASAN-crashing proof in this lab.
- 2026-05-21: Added a raw HTTP/2 upstream generator and
  `tools/poolslip_h2_upstream_probe.py`. The upstream rotates through controlled
  H2 frame sequences: valid headers, valid data, 103 then final, duplicate
  status, missing status, split HEADERS/CONTINUATION, invalid HPACK, DATA before
  HEADERS, large response header, malformed padding, RST after headers, and
  GOAWAY after headers. Next step is live ASAN execution against the rebuilt H2
  lab.
- 2026-05-21: Ran
  `tools/poolslip_h2_upstream_probe.py --target 127.0.0.1:19361 --iterations
  120 --timeout 5 --container nginx-h2-header-1310-amd64-asan
  --stop-on-suspicious` against the `release-1.31.0` H2 ASAN lab. Result:
  `summary suspicious=0 iterations=120`, `asan_log_bytes 0`, `asan_status
  clean`. The parser returned expected `200`, `502`, or empty-body close
  outcomes, health remained up, and no sanitizer finding was produced. This is
  negative evidence for the first `ngx_http_proxy_v2_module` frame-parser
  hypotheses; it does not close deeper HPACK/dynamic-table or flow-control
  state-space yet.
